# Operator Core + OriginPool CRD — Design Spec

**Sub-project**: 2 of 5 (Operator Core + OriginPool CRD)
**Date**: 2026-04-20
**Status**: Approved

## Overview

A Kubebuilder-based Kubernetes operator that reconciles OriginPool custom resources against the F5 Distributed Cloud REST API. This is the first end-to-end reconciliation loop — it establishes the operator scaffolding, shared utilities, and patterns that sub-projects 3–5 will reuse for additional resource types and higher-level integrations.

## Scope

### In Scope

- Kubebuilder project scaffold with API group `xc.f5.com/v1alpha1`
- OriginPool CRD with spec mirroring XC API structure
- Reconciler implementing Get→Compare→Act loop via `internal/xcclient`
- Finalizer-based deletion with orphan annotation escape hatch
- Status conditions (`Ready`, `Synced`) and XC metadata in status
- Credential loading from Kubernetes Secret (API token or P12 mTLS)
- Secret watch for credential rotation with mutex-protected client swap
- XC namespace resolution: K8s namespace default, annotation override
- Operator configuration via CLI flags
- Unit tests (fake XC client), envtest integration tests (fake XC server), contract tests (real cluster + real API)

### Out of Scope

- Additional CRDs beyond OriginPool (sub-project 3)
- Ingress controller or Gateway API integration (sub-projects 4–5)
- Multi-tenant support (different XC tenants per namespace)
- Webhook validation (can be added later)
- OLM bundle or marketplace distribution
- Config CRD for runtime reconfiguration of operator settings

### Component Relationships

The manager binary (`cmd/main.go`) wires everything together:

1. Parses CLI flags (tenant URL, Secret reference, rate limits)
2. Reads the credential Secret, builds initial `xcclient.Client`
3. Wraps it in a `xcclientset.ClientSet` (mutex-protected, hot-swappable)
4. Sets up a Secret informer that triggers client re-initialization on change
5. Creates the OriginPool controller, injects the `ClientSet`
6. Starts the manager (leader election, health probes, metrics)

Sub-projects 3–5 add more controllers in step 5 — the rest stays the same.

## CRD Types

### API Group and Version

- **Group:** `xc.f5.com`
- **Version:** `v1alpha1`
- **Kind:** `OriginPool`
- **Plural:** `originpools`
- **Short name:** `op`

### OriginPool Spec

CRD spec fields mirror the XC API 1:1. Field names use camelCase (K8s convention) — the controller translates to snake_case when building XC client requests.

```go
type OriginPoolSpec struct {
    OriginServers         []OriginServer        `json:"originServers"`
    Port                  int                   `json:"port"`
    LoadBalancerAlgorithm string                `json:"loadBalancerAlgorithm,omitempty"`
    HealthChecks          []ObjectRef           `json:"healthChecks,omitempty"`
    UseTLS                *apiextensionsv1.JSON `json:"useTLS,omitempty"`
    NoTLS                 *apiextensionsv1.JSON `json:"noTLS,omitempty"`
}

type OriginServer struct {
    PublicIP      *PublicIP      `json:"publicIP,omitempty"`
    PublicName    *PublicName    `json:"publicName,omitempty"`
    PrivateIP     *PrivateIP     `json:"privateIP,omitempty"`
    PrivateName   *PrivateName   `json:"privateName,omitempty"`
    K8SService    *K8SService    `json:"k8sService,omitempty"`
    ConsulService *ConsulService `json:"consulService,omitempty"`
}

type PublicIP struct {
    IP string `json:"ip"`
}

type PublicName struct {
    DNSName string `json:"dnsName"`
}

type PrivateIP struct {
    IP   string     `json:"ip"`
    Site *ObjectRef `json:"site,omitempty"`
}

type PrivateName struct {
    DNSName string     `json:"dnsName"`
    Site    *ObjectRef `json:"site,omitempty"`
}

type K8SService struct {
    ServiceName      string     `json:"serviceName"`
    ServiceNamespace string     `json:"serviceNamespace,omitempty"`
    Site             *ObjectRef `json:"site,omitempty"`
}

type ConsulService struct {
    ServiceName string     `json:"serviceName"`
    Site        *ObjectRef `json:"site,omitempty"`
}

type ObjectRef struct {
    Name      string `json:"name"`
    Namespace string `json:"namespace,omitempty"`
    Tenant    string `json:"tenant,omitempty"`
}
```

**TLS configuration:** `UseTLS` and `NoTLS` are opaque JSON (`apiextensionsv1.JSON`) because the OneOf variants are complex and rarely managed by the operator. Power users can pass the full XC API TLS config structure directly.

### OriginPool Status

```go
type OriginPoolStatus struct {
    Conditions         []metav1.Condition `json:"conditions,omitempty"`
    ObservedGeneration int64              `json:"observedGeneration,omitempty"`
    XCResourceVersion  string             `json:"xcResourceVersion,omitempty"`
    XCUID              string             `json:"xcUID,omitempty"`
    XCNamespace        string             `json:"xcNamespace,omitempty"`
}
```

**Conditions:**

| Type | Meaning |
|------|---------|
| `Ready` | Overall health — True when XC resource exists and matches desired state |
| `Synced` | Last reconciliation result — True on success, False on failure with reason |

**Condition reasons:**

| Reason | Condition | Meaning |
|--------|-----------|---------|
| `CreateSucceeded` | Synced=True | XC resource created |
| `UpdateSucceeded` | Synced=True | XC resource updated |
| `UpToDate` | Synced=True | No changes needed |
| `DeleteSucceeded` | Synced=True | XC resource deleted |
| `CreateFailed` | Synced=False | Create call failed |
| `UpdateFailed` | Synced=False | Replace call failed |
| `DeleteFailed` | Synced=False | Delete call failed |
| `AuthFailure` | Ready=False | XC API returned 401/403 |
| `RateLimited` | Synced=False | XC API returned 429, will retry |
| `ServerError` | Synced=False | XC API returned 5xx, will retry |
| `Conflict` | Synced=False | XC API returned 409, will retry |

### Annotations

| Annotation | Purpose | Default |
|------------|---------|---------|
| `f5xc.io/namespace` | Override XC namespace (instead of K8s namespace) | K8s namespace |
| `f5xc.io/deletion-policy` | Set to `orphan` to keep XC resource on CR deletion | delete |

### Finalizer

```
finalizers:
  - xc.f5.com/cleanup
```

Added on first reconcile, removed after successful XC deletion (or immediately if deletion-policy is orphan).

## Reconciliation Loop

### Main Flow

```
Reconcile(ctx, req):
  1. Fetch OriginPool CR from K8s cache
     - Gone (not found) → return success (nothing to do)
  2. Check deletionTimestamp
     - Set → run deletion flow (see below)
  3. Ensure finalizer present
     - Missing → add finalizer, return (requeue triggered by update)
  4. Resolve XC namespace
     - Check f5xc.io/namespace annotation
     - Fall back to CR's K8s namespace
  5. Get current XC resource
     - Call GetOriginPool(ctx, xcNamespace, cr.Name)
  6. If ErrNotFound → Create flow
     a. Build OriginPoolCreate from CR spec
     b. Call CreateOriginPool
     c. Update status: Ready=True, Synced=True (CreateSucceeded)
     d. Populate xcResourceVersion, xcUID, xcNamespace
  7. If exists → Update flow
     a. Build desired spec as JSON
     b. Call ClientNeedsUpdate(current.RawSpec, desiredJSON)
     c. If no update needed:
        - Update status: Synced=True (UpToDate)
        - Return (no requeue)
     d. If update needed:
        - Build OriginPoolReplace with resource_version from GET
        - Call ReplaceOriginPool
        - Update status: Ready=True, Synced=True (UpdateSucceeded)
        - Populate xcResourceVersion from response
  8. Error handling (any step):
     - ErrConflict → requeue immediately
     - ErrRateLimited → requeue with backoff
     - ErrAuth → set Ready=False (AuthFailure), do NOT requeue
     - ErrServerError → requeue with delay (30s)
     - Other errors → requeue with delay (10s)
```

### Deletion Flow

```
handleDeletion(ctx, cr):
  1. Check f5xc.io/deletion-policy annotation
  2. If not "orphan":
     a. Resolve XC namespace
     b. Call DeleteOriginPool(ctx, xcNamespace, cr.Name)
     c. ErrNotFound → treat as success (already gone)
     d. Other errors → return error (retry deletion)
  3. Remove finalizer from CR
  4. Return success
```

### Spec Translation

Two functions convert between K8s CRD types and XC client types:

- `buildOriginPoolCreate(cr *v1alpha1.OriginPool) *xcclient.OriginPoolCreate` — maps camelCase CRD fields to snake_case XC structs, sets metadata.name and metadata.namespace
- `buildOriginPoolReplace(cr *v1alpha1.OriginPool, resourceVersion string) *xcclient.OriginPoolReplace` — same mapping plus resource_version for optimistic concurrency

These are pure functions with no side effects — straightforward to unit test.

### Requeue Strategy

| Scenario | Requeue |
|----------|---------|
| Success (create/update/no-op) | No requeue — rely on watch events |
| ErrConflict | Immediate (`ctrl.Result{Requeue: true}`) |
| ErrRateLimited | 60 seconds |
| ErrServerError | 30 seconds |
| Other transient errors | 10 seconds |
| ErrAuth | No requeue (broken credentials won't self-heal) |

## Credential Management

### Secret Format

The operator reads a Kubernetes Secret referenced by CLI flags. Auth method is auto-detected by which keys are present.

**API Token auth:**

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: xc-credentials
type: Opaque
data:
  api-token: <base64-encoded token>
```

**P12 mTLS auth:**

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: xc-credentials
type: Opaque
data:
  cert.p12: <base64-encoded P12 file>
  cert-password: <base64-encoded password>
```

If both `api-token` and `cert.p12` are present, the operator fails with a clear error — same mutual exclusion as `xcclient.Config`.

### Credential Rotation

The manager watches the credential Secret via an informer. On change:

1. Read new Secret data
2. Build new `xcclient.Config`, validate it
3. Construct new `xcclient.Client`
4. Swap into the `xcclientset.ClientSet` behind a `sync.RWMutex`

If the new Secret is invalid (missing keys, bad P12, validation failure), the operator logs an error and keeps the existing client — no downgrade to a broken state.

### internal/credentials/loader.go

```go
func ConfigFromSecret(secret *corev1.Secret, tenantURL string, rateLimits xcclient.RateLimitConfig) (xcclient.Config, error)
```

Reads Secret data, detects auth method, returns a populated `xcclient.Config`. For P12 auth, the function writes the P12 bytes to a temp file and sets `CertP12Path` — `xcclient.Config` requires a file path, not raw bytes. The caller is responsible for temp file cleanup. Pure function apart from the temp file write — no K8s client calls.

### internal/xcclientset/clientset.go

```go
type ClientSet struct {
    mu     sync.RWMutex
    client xcclient.XCClient
}

func (cs *ClientSet) Get() xcclient.XCClient     // Read lock, return current client
func (cs *ClientSet) Swap(c xcclient.XCClient)    // Write lock, replace client
```

The reconciler calls `cs.Get()` at the start of each reconciliation. Concurrent reconciliations using the old client finish normally — only new reconciliations pick up the swapped client.

## Operator Configuration

### CLI Flags

| Flag | Required | Default | Description |
|------|----------|---------|-------------|
| `--tenant-url` | Yes | — | F5 XC API base URL |
| `--credentials-secret` | Yes | — | Name of Secret containing auth credentials |
| `--credentials-secret-namespace` | No | Operator's namespace | Namespace of the credential Secret |
| `--rate-limit-rps` | No | 2 | Default requests per second per XC API endpoint |
| `--rate-limit-burst` | No | 5 | Burst allowance per endpoint |
| `--metrics-bind-address` | No | `:8080` | Prometheus metrics endpoint |
| `--health-probe-bind-address` | No | `:8081` | Health/readiness probe endpoint |
| `--leader-elect` | No | false | Enable leader election for HA |

### Manager Startup Sequence

```
1. Parse CLI flags
2. Read credential Secret
3. Build xcclient.Config from Secret + flags
4. Construct xcclient.Client
5. Wrap in xcclientset.ClientSet
6. Create controller-runtime Manager
7. Set up Secret watch → on change, rebuild client and Swap
8. Register OriginPool controller with Manager
9. Start Manager (blocks until shutdown)
```

## Testing Strategy

### Unit Tests (no cluster, no network)

**Reconciler tests** (`internal/controller/originpool_controller_test.go`):

Use a fake implementation of `xcclient.XCClient` (hand-written or generated mock) to test reconciler logic in isolation:

- GET returns ErrNotFound → reconciler calls Create
- GET returns object, NeedsUpdate=true → reconciler calls Replace
- GET returns object, NeedsUpdate=false → reconciler skips Replace
- Each error type → correct status condition and requeue behavior
- Deletion with finalizer → Delete called, finalizer removed
- Deletion with orphan annotation → Delete NOT called, finalizer removed
- XC namespace annotation override works correctly
- Missing finalizer → added on first reconcile

**Spec translation tests:**
- `buildOriginPoolCreate` maps all CRD fields correctly
- `buildOriginPoolReplace` includes resource_version
- camelCase → snake_case mapping is correct for all OriginServer types

**Credential loader tests** (`internal/credentials/loader_test.go`):
- Secret with `api-token` → Config with APIToken set
- Secret with `cert.p12` + `cert-password` → Config with CertP12Path (written to temp file)
- Secret with both → error
- Secret with neither → error
- Missing `cert-password` with `cert.p12` → error

**ClientSet tests** (`internal/xcclientset/clientset_test.go`):
- Concurrent Get() calls return same client
- Swap() changes what Get() returns
- In-flight Get() during Swap() gets either old or new (no panic)

Run with: `make test`

### Integration Tests (envtest + fake XC server)

Wire the reconciler to:
- **envtest** for the K8s side (real etcd + apiserver, CRD installed)
- **FakeXCServer** from `internal/xcclient/testutil` for the XC API side

Test scenarios:
- Apply OriginPool CR → verify FakeXCServer received Create call → verify CR status is Ready=True, Synced=True
- Update CR spec → verify FakeXCServer received Replace call → verify status updated
- Delete CR → verify FakeXCServer received Delete call → verify CR is gone
- Inject 429 from FakeXCServer → verify status shows RateLimited, reconciler requeues
- Inject 409 → verify reconciler retries
- Inject 401 → verify Ready=False with AuthFailure reason, no requeue

Run with: `make test`

### Contract Tests (real cluster + real XC API)

Build tag: `contract`. Require a real K8s cluster (kubeconfig access) and XC credentials in environment.

- Install CRD in cluster
- Apply OriginPool CR
- Wait for Ready=True
- Verify resource exists in XC API via direct client call
- Update CR spec
- Wait for status update
- Verify XC resource changed
- Delete CR
- Verify XC resource is gone

Run with: `make test-contract`
